<objective>
Build ComponentDetector and ConventionDetector — the first two of four detectors that provide codebase intelligence for AI commit message generation.

ComponentDetector maps changed file paths to logical components (monorepo packages, services, modules). It extends (not replaces) the existing `project-type-detector.js` for monorepo/WordPress detection per D-01, supports custom component mappings via config per D-02, and auto-detects common directory patterns.

ConventionDetector analyzes project conventions through lightweight heuristics — file name sampling and regex on file contents — per D-03. It detects naming patterns, file structure patterns, and import/export conventions.

Purpose: Enrich AI commit prompts with "which component changed" and "what conventions does this project follow" context.
Output: Two tested detector modules with flat context object outputs.
</objective>

  <name>Task 1: Create ComponentDetector (DET-01)</name>
  <files>src/detectors/component-detector.js, tests/detectors/component-detector.test.js</files>
  <behavior>
    - Given file paths ['packages/auth/src/login.js', 'packages/auth/src/logout.js'], detect returns { component: 'auth', scope: 'packages/auth', boundary: 'package' }
    - Given file paths ['services/user-service/handler.js'], detect returns { component: 'user-service', scope: 'services/user-service', boundary: 'directory' }
    - Given file paths ['src/auth/login.js'] in a non-monorepo project, detect returns { component: 'auth', scope: 'src/auth', boundary: 'directory' }
    - Given custom config { componentMap: { 'web/': 'frontend-app', 'api/': 'backend-api' } }, files in 'web/' map to 'frontend-app'
    - Custom config mappings take priority over auto-detected patterns
    - When no component detected, returns { component: null, scope: null, boundary: null }
    - When repo is monorepo (package.json has workspaces or lerna.json exists), uses package.json names from detected package dirs
    - Handles edge case: file at repo root (no parent directory to extract component from)
    - Handles edge case: deeply nested path extracts nearest meaningful component
  </behavior>
  <action>
    Create `src/detectors/component-detector.js` as a class `ComponentDetector`:

    1. **Constructor:** Accept `repoRoot` (string). Create the `src/detectors/` directory if it doesn't exist.

    2. **`detect(filePaths)`** — Main method. Accepts `filePaths: string[]`. Returns flat object:
       ```javascript
       { component: string|null, scope: string|null, boundary: string|null }
       ```
       - `boundary` values: `'package'` (monorepo package.json), `'directory'` (common dir pattern), `'config'` (custom mapping), `null`
       - Logic:
         a. Read custom componentMap from config (use `conf` package, key: `componentMap`). Format: `{ 'path/prefix': 'component-name' }`. Check each filePath against config prefixes first (longest prefix match wins). If match found → `{ component, scope: prefix, boundary: 'config' }`.
         b. Check if monorepo: call `ProjectTypeDetector.detectMonorepo(this.repoRoot)`. If monorepo, find which package each file belongs to by checking for `package.json` in parent directories up to repoRoot. Extract `name` field from that package.json → `{ component: name, scope: relativePath, boundary: 'package' }`.
         c. Auto-detect common patterns: check filePath segments against `['packages/', 'apps/', 'services/', 'modules/', 'libs/', 'components/']`. If match → extract next segment as component name → `{ component, scope, boundary: 'directory' }`.
         d. Fallback: extract first meaningful directory from filePath path segments (skip common prefixes like `src/`, `lib/`) → `{ component, scope, boundary: 'directory' }`.
         e. If nothing found → return `{ component: null, scope: null, boundary: null }`.

    3. **`detectBatch(filePaths)`** — Accepts array, calls detect for each, returns array of results. Used for processing all changed files at once.

    4. **`getComponentsSummary(filePaths)`** — Calls detectBatch, then aggregates: returns `{ components: string[], filesByComponent: { [component]: string[] } }`.

    Use `require('../utils/project-type-detector')` for `ProjectTypeDetector`. Use `require('conf')` for config access (same package already used in `config-manager.js`). Use CommonJS `module.exports = ComponentDetector`.

    Per D-01: Extend existing project-type-detector.js, not replace it.
    Per D-02: Config key is `componentMap` under `~/.config/ai-commit-generator/config.json`.
    Per D-06: Return flat context object, no wrappers/metadata/confidence/timestamps.
  </action>
  <verify>
    <automated>npx jest tests/detectors/component-detector.test.js --verbose</automated>
  </verify>

<task type="auto" tdd="true">
  <name>Task 2: Create ConventionDetector (DET-02)</name>
  <files>src/detectors/convention-detector.js, tests/detectors/convention-detector.test.js</files>
  <behavior>
    - Given files ['getUser.js', 'setUser.js', 'deleteUser.js'], naming is 'camelCase'
    - Given files ['get_user.py', 'set_user.py'], naming is 'snake_case'
    - Given files ['get-user.js', 'set-user.js'], naming is 'kebab-case'
    - Given files mixed naming conventions, returns the dominant one (most frequent)
    - Given file structure ['src/features/auth/login.js', 'src/features/auth/register.js', 'src/features/payments/checkout.js'], structure is 'feature-based'
    - Given file structure ['src/controllers/auth.js', 'src/models/user.js', 'src/utils/helper.js'], structure is 'layered'
    - Given file structure with flat files at root, structure is 'flat'
    - Given file contents with `require('./utils')` and `require('../models/user')`, imports are 'relative'
    - Given file contents with `require('@company/auth')` or `require('/src/utils')`, imports are 'absolute'
    - Returns flat object: { naming: string, structure: string, imports: string }
    - When repo has no source files, returns { naming: 'unknown', structure: 'unknown', imports: 'unknown' }
  </behavior>
  <action>
    Create `src/detectors/convention-detector.js` as a class `ConventionDetector`:

    1. **Constructor:** Accept `repoRoot` (string).

    2. **`analyze()`** — Main method. Accepts no args (reads from repoRoot). Returns flat object:
       ```javascript
       { naming: string, structure: string, imports: string }
       ```
       - `naming` values: `'camelCase'`, `'snake_case'`, `'kebab-case'`, `'PascalCase'`, `'mixed'`, `'unknown'`
       - `structure` values: `'feature-based'`, `'layered'`, `'nested'`, `'flat'`, `'unknown'`
       - `imports` values: `'relative'`, `'absolute'`, `'mixed'`, `'unknown'`

    3. **Naming detection** — Sample file names from the repository (use `fs.readdirSync` recursively, limit to ~50 files for speed per D-03 <100ms target):
       - Extract base filenames (without extension)
       - Count occurrences of each pattern:
         - camelCase: `/^[a-z][a-zA-Z0-9]*$/`
         - snake_case: `/^[a-z][a-z0-9_]*$/` (contains underscore)
         - kebab-case: `/^[a-z][a-z0-9-]*$/` (contains hyphen)
         - PascalCase: `/^[A-Z][a-zA-Z0-9]*$/`
       - Return the dominant pattern (>50% of files). If no dominant → `'mixed'`. If no files → `'unknown'`.

    4. **Structure detection** — Analyze directory tree depth and organization:
       - Read top-level dirs under `src/` (or repoRoot if no src/)
       - If dirs match feature names (multiple parallel dirs with similar substructure like `components/`, `pages/`, `hooks/` inside each) → `'feature-based'`
       - If dirs match layer names (`controllers/`, `models/`, `views/`, `services/`, `utils/`) → `'layered'`
       - If 3+ nesting levels with varied structure → `'nested'`
       - If most files at top level → `'flat'`
       - Otherwise → `'unknown'`

    5. **Import convention detection** — Scan `.js` file contents for import patterns:
       - Sample ~20 `.js` files (first 20 found via recursive walk)
       - Count patterns in file contents:
         - Relative: `/require\s*\(\s*['"]\.\.?\//` or `/from\s+['"]\.\.?\//`
         - Absolute: `/require\s*\(\s*['"]@/` or `/require\s*\(\s*['"][\/]src/` or `/from\s+['"]@/`
       - Return dominant pattern (>70% threshold). If neither dominates → `'mixed'`. If no JS files → `'unknown'`.

    6. **`analyzeFiles(filePaths)`** — Accepts specific file paths, analyzes only those files. Same return type. Used for analyzing changed files specifically.

    Use `require('fs-extra')` and `require('path')` (already project deps). CommonJS `module.exports = ConventionDetector`.

    Per D-03: Lightweight heuristics, no AST, target <100ms execution, no new dependencies.
    Per D-06: Return flat context object.
  </action>

  </verify>
