Build FileTypeDetector and DependencyMapper — the remaining two detectors that provide file classification and code relationship awareness for AI commit message generation.

FileTypeDetector categorizes files by language, type (source/config/docs/assets), and framework. It extends the categorization already present in `analysis-engine.js`'s `categorizeFiles()` method, adding per-file type detection and framework identification.

Purpose: Enrich AI commit prompts with "what types of files changed" and "what other files depend on the changes" context.
Output: Two tested detector modules with flat context object outputs.

  <name>Task 2: Create DependencyMapper (DET-04)</name>
  <files>src/detectors/dependency-mapper.js, tests/detectors/dependency-mapper.test.js</files>
  <behavior>
    - Given file content "const utils = require('./utils')", parseImports returns [{ module: './utils', type: 'commonjs', resolvedPath: null }]
    - Given file content "import { foo } from '../lib/bar'", parseImports returns [{ module: '../lib/bar', type: 'esm', resolvedPath: null }]
    - Given file content with both require and import, both are detected
    - Given file content "const x = require(dynamicPath)", dynamic requires are skipped (not statically analyzable)
    - Given file content with template literals in require(`./modules/${name}`), skipped as dynamic
    - Given a set of files and their contents, mapDependencies returns a graph with imports and exports
    - Given file A imports file B, and file B imports file C, findDependents('B') returns ['A'] only (NOT C — direct dependents per D-05)
    - mapDependencies returns { imports: [...], exports: [...], affected: [...] }
    - Handles edge case: file with no imports/exports returns empty arrays
    - Handles edge case: circular dependency detected and flagged (not followed recursively)
  </behavior>
  <action>
    Create `src/detectors/dependency-mapper.js` as a class `DependencyMapper`:

    1. **Constructor:** Accept `repoRoot` (string).

    2. **`parseImports(fileContent)`** — Accepts `fileContent: string`. Returns array:
       ```javascript
       Array<{ module: string, type: 'commonjs'|'esm'|'dynamic', resolvedPath: string|null }>
       ```
       - CommonJS patterns (regex):
         - `require\s*\(\s*['"]([^'"]+)['"]\s*\)` — static require
       - ESM patterns (regex):
         - `import\s+.*?\s+from\s+['"]([^'"]+)['"]` — named/default import
         - `import\s+['"]([^'"]+)['"]` — side-effect import
         - `export\s+.*?\s+from\s+['"]([^'"]+)['"]` — re-export
       - Dynamic patterns (SKIP these — not statically analyzable per agent's discretion):
         - `require\s*\(` followed by anything other than a string literal
         - Template literals: `require(\`...\`)`
       - `resolvedPath` is always `null` at parse time (resolution is separate step)
       - Skip node_modules imports (modules not starting with `.` or `/`)

    3. **`parseExports(fileContent)`** — Accepts `fileContent: string`. Returns array:
       ```javascript
       Array<{ name: string, type: 'named'|'default'|'reexport' }>
       ```
       - Named: `module\.exports\s*=\s*(\w+)` or `exports\.(\w+)\s*=` or `export\s+(?:const|let|var|function|class)\s+(\w+)`
       - Default: `module\.exports\s*=` (when not assigning a named thing) or `export\s+default`
       - Re-export: `export\s+\{.*\}\s+from`

    4. **`mapDependencies(filesWithContents)`** — Accepts `filesWithContents: Map<string, string>` (absolute or relative filePath → content). Returns flat object:
       ```javascript
       {
         imports: Array<{ file: string, module: string, type: string }>,
         exports: Array<{ file: string, name: string, type: string }>,
         affected: string[]  // files that import any of the changed files (direct dependents only)
       }
       ```
       - Build an internal graph: for each file, parse its imports and resolve relative paths against repoRoot
       - `affected`: given all files in `filesWithContents`, find ALL other `.js` files in the repo that import any of them. This requires scanning repo source files for require/import of changed file paths. Implementation: build a reverse lookup map from all source files, then check which ones reference changed files. For efficiency, limit scan to `src/` directory. Skip node_modules.
       - Mark circular dependencies: if A imports B and B imports A, note in output but don't follow recursively per D-05.

    5. **`findDependents(changedFilePath, repoSourceFiles)`** — Accepts changed file path and optional list of repo source files to scan. Returns:
       ```javascript
       Array<{ file: string, importStatement: string }>
       ```
       - Scan repo source files for `require('./path/to/changed')` or `import from './path/to/changed'`
       - Match by file path (normalize both to handle `./` vs `../` vs absolute)
       - Direct dependents ONLY — do not recurse (per D-05)

    Use `require('fs-extra')` and `require('path')` (already project deps). CommonJS `module.exports = DependencyMapper`.

    Per D-04: Regex-based parsing for import/export statements. Handles CommonJS and ESM.
    Per D-05: Direct dependents only, no transitive closure.
    Per D-06: Return flat context objects.
  </action>
